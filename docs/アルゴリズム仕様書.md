# 自動スケジューリングアルゴリズム仕様書

作成日: 2026-04-11
プロジェクト: shift-create-app
対象: Phase 1 自動シフト生成機能

---

## 1. アルゴリズム選定

### 採用: Google OR-Tools CP-SAT（制約充足ソルバー）

| 比較項目 | CP-SAT | ヒューリスティック（貪欲法・遺伝的アルゴリズム） |
|---------|--------|------------------------------------------------|
| 制約の網羅性 | ハード制約を確実に満たす（充足不可能なら明示的にエラー） | 制約を「なるべく」守る設計が多く、見落としリスクあり |
| 実装コスト | 制約を宣言的に書くだけ | チューニングが複雑・経験が必要 |
| 品質保証 | 解が見つかれば最適解または最良近似解が保証される | 解の品質が不安定 |
| スケール | 30人 × 31日 × シフト種別5〜8 → 十分実用的な速度 | 高速だが品質保証なし |
| 将来拡張性 | 制約追加が容易 | 追加ごとにロジック全体を見直す必要あり |

**結論**: 病棟シフトはハード制約（労基法・夜勤明け・最低配置数）が絶対条件であり、CP-SAT が最適。

## 3. 問題の定式化

### 3.1 変数

```python
# shifts[s][d][t] ∈ {0, 1}
# s: staff_id のインデックス
# d: 日付インデックス (0 〜 days_in_month-1)
# t: shift_type_id のインデックス
shifts[(s, d, t)] = model.new_bool_var(f"shift_s{s}_d{d}_t{t}")
```

### 3.2 ハードコンストレイント（必ず満たす）

| # | 制約名 | 内容 |
|---|--------|------|
| H1 | 1日1シフト | 1スタッフにつき、1日に割り当てるシフトは1種別のみ |
| H2 | 最低配置人数 | 各日・各シフト種別で `shift_constraints.min_staff_per_shift` を満たす |
| H3 | 休暇希望の反映 | `leave_requests.type` が `希望休` / `有給` / `特別休暇` の日は、勤務シフトを割り当てない |
| H4 | 夜勤翌日の明け自動挿入 | `is_overnight = true` のシフト翌日は、`is_off = true` のシフト（明け）を自動割り当て。**月末越えの場合（例: 5/31夜勤 → 6/1明け）は翌月の1日目に明けを挿入する。翌月の割り当ては次回シフト生成時に前月末の夜勤情報を引き継いで反映する（リクエストに `prev_month_last_night_shifts` として前月末の夜勤スタッフIDリストを渡す）。** |
| H5 | 最大連続勤務日数 | `shift_constraints.max_consecutive_work_days` を超えて連続勤務させない |
| H6 | 夜勤後の最低休息 | `min_rest_hours_after_night` で定義された休息時間を確保（H4と連動） |
| H7 | must_pair 制約 | `staff_pair_constraints.constraint_type = 'must_pair'` のペアは、対象シフト種別に必ず同じ日に配置 |
| H8 | must_not_pair 制約 | `staff_pair_constraints.constraint_type = 'must_not_pair'` のペアは、対象シフト種別に同日配置しない |
| H9 | 月間最大夜勤回数 | `shift_constraints.max_night_shifts_per_month` を超えて夜勤を割り当てない |

### 3.3 ソフトコンストレイント（目的関数で最大化・最小化）

| # | 制約名 | 内容 | 重み（暫定） |
|---|--------|------|-------------|
| S1 | 夜勤回数の均等化 | 各スタッフの夜勤回数をターゲット値（総夜勤数 ÷ スタッフ数）に近づける。**ターゲット値からの偏差絶対値合計を最小化**（最大値−最小値方式は CP-SAT で補助変数が必要なため採用しない） | weight = 100 |
| S2 | 月間勤務時間の平準化 | スタッフ間の月間勤務時間の差を最小化 | weight = 50 |
| S3 | スキルバランス（経験者配置） | `skill_shift_requirements.min_experienced_count` を満たす経験者数を最大化 | weight = 200 |
| S4 | 休暇希望の優先充足 | 登録された希望休・有給・特別休暇をできるだけ全件反映（H3でハード化しているため、Phase 1 では H3 として扱う） | - |
| S5 | シフト希望の反映 | `leave_requests.type = 'シフト希望'` の日に `preferred_shift_type_id` のシフトをできるだけ割り当てる | weight = 150 |

**目的関数**:
```
maximize(
  S3_score * 200
  + S5_score * 150           # シフト希望（夜勤希望など）の充足数
  - S1_deviation_sum * 100   # 各スタッフの夜勤回数とターゲット値の偏差絶対値合計
  - S2_penalty * 50
)
```

### ステータス一覧

| status | 意味 |
|--------|------|
| `success` | 最適解または十分な近似解が見つかった |
| `infeasible` | ハード制約を満たす解が存在しない |
| `timeout` | タイムアウト内に十分な解が見つからなかった（部分解を返す場合あり） |
| `error` | 入力データの不整合・サーバーエラー |

## 5. タイムアウトと品質トレードオフ

| タイムアウト | 期待品質 | 推奨シナリオ |
|------------|---------|-------------|
| 10秒 | 十分実用的 | スタッフ20名以下・制約が少ない |
| 30秒 | 高品質 | スタッフ30名・全制約あり（Phase 1 デフォルト） |
| 60秒 | 最高品質 | 手動で最適化を要求する場合（Phase 3） |

**Phase 1 デフォルト**: `timeout_seconds = 30`

タイムアウト時は、その時点の最良解を返す（`status: "timeout"`）。管理者はシフト表を手動編集して対応する。

## 6. エラーハンドリング

| 状況 | 処理 |
|------|------|
| 対象月のスタッフが0人 | `error` を返す（入力バリデーションで検出） |
| 最低配置人数 > 在籍スタッフ数 | `infeasible` を返す（ソルバー実行前に検出推奨） |
| `must_pair` のスタッフ双方が希望休 | `infeasible` を返す |
| ソルバー内部エラー | `error` を返す（ログに詳細記録） |
| タイムアウト（解なし） | `timeout` を返す（assignments は空配列） |
| タイムアウト（部分解あり） | `timeout` を返す（assignments に部分解を含む） |

## 7. Pythonサービス実装方針

### ディレクトリ構成（予定）

```
scheduler-service/
├── main.py              # FastAPI エントリポイント
├── solver/
│   ├── __init__.py
│   ├── model.py         # CP-SAT モデル定義
│   ├── constraints.py   # ハード制約の追加
│   ├── objectives.py    # 目的関数（ソフト制約）
│   └── parser.py        # リクエスト → ソルバー入力変換
├── schemas.py           # Pydantic スキーマ
├── Dockerfile
└── requirements.txt
```

### 主要実装ポイント

1. **制約の追加順序**: ハード制約 → ソフト制約（目的関数）の順で追加
2. **タイムアウト設定**: `solver.parameters.max_time_in_seconds = timeout_seconds`
3. **ヒント（hints）の活用**: 前回生成結果をヒントとして渡すことで収束を高速化（Phase 3）
4. **ログ**: ソルバーの統計情報（探索ノード数・解の数・目的関数値）をCloudLoggingに出力

## 9. テスト方針

| テスト種別 | 内容 |
|-----------|------|
| ユニットテスト | 各制約が正しく追加されているかを単体検証（pytest） |
| 統合テスト | サンプルデータでエンドツーエンド生成テスト（全ハード制約を検証） |
| 境界テスト | ・スタッフ1名 ・希望休が全日 ・最低配置数 = スタッフ数 などの極端ケース |
| 充足不可テスト | 意図的に充足不可能な制約を与え `infeasible` が返ることを確認 |
| パフォーマンステスト | 30名 × 31日 × 8シフト（最大規模）で **30秒以内** に `success` または `timeout`（部分解あり）が返ることを確認 |
