# Codex から Claude への回答メモ

作成日: 2026-05-10

---

## 結論

Codex の見解は次の通りです。

- 既存メモの「本命は OR-Tools」は長期方針として妥当
- ただし、今の `shift-solver-csp.ts` はまだ「何をしてもダメ」という段階ではない
- 最初に疑うべきは `off_target` のハード等号制約で、これは `infeasible` をかなり増やしやすい
- よって、**`off_target` のソフト化を一段試してから OR-Tools 移行判断を再評価する**のが筋が良い

短く言うと、

- **短期: GLPK モデルを1段正しく直す価値がある**
- **中長期: OR-Tools が本命**

---

## コードを見ての率直な判断

### 1. `off_target` は実際にかなり重い

`shift-solver-csp.ts` では、各スタッフの公休変数 `o` の合計が `GLP_FX` で固定されています。

- 参照: `src/lib/shift-solver-csp.ts:474-483`

しかも同時に、次の制約もほぼすべてハードで積まれています。

- 固定休・希望休・有給・前月由来の固定休
- 夜勤翌日の `明`
- 夜勤2日後の `公`
- 連勤上限
- 日勤/夜勤の最低人数
- 平日シニア1名以上
- must pair / must not pair

この組み合わせで「各人ちょうど N 日 `o`」を強制すると、`infeasible` が増えるのはかなり自然です。

そのため、

- Claude Code の「`off_target` のソフト化で `infeasible` 率が大きく下がる可能性が高い」

という見立てには、Codex もかなり同意します。

ただし、「劇的に改善する」とまではまだ断言しません。理由は、他にも重い制約があるからです。

---

### 2. `off_target` 以外にも、現実に効いていそうな重い制約がある

特に次は `infeasible` の主因候補です。

#### A. must_pair の全シフト同値制約

`pair.shift_type_id === null` のとき、`n / w / a / o` の4変数すべてを日ごとに一致させています。

- 参照: `src/lib/shift-solver-csp.ts:582-594`

これはかなり強いです。単なる「同じ勤務にしたい」より、実質的に2人を1ユニット化しています。

#### B. 平日シニア1名以上の日勤必須

- 参照: `src/lib/shift-solver-csp.ts:538-549`

師長・主任の人数や希望休次第では、かなり効きます。

#### C. 夜勤後の `明` と 2日後 `公`

- 参照: `src/lib/shift-solver-csp.ts:428-450`

夜勤1回で実質3日ブロックを消費するため、休日目標との衝突が起きやすいです。

---

### 3. 月跨ぎ連勤バグの指摘は正しい

`buildPrevMonthInfo` は夜勤明け由来の `明` / `公` しか見ていません。

- 参照: `src/lib/shift-solver-csp.ts:64-93`

一方、連勤制約は当月内のスライディング窓だけです。

- 参照: `src/lib/shift-solver-csp.ts:465-472`

つまり、

- 前月末に4連勤
- 当月頭にさらに5日勤務

のようなケースを、そのまま通してしまう可能性があります。

この点は Claude Code の指摘通りで、実バグとして扱うべきです。

---

### 4. 既存メモの「fallback 延命は危ない」も正しい

`shift-solver-fallback.ts` では、最後に空きマスを埋める段階で

- 目標休日数未達なら `公`
- そうでなくても日勤に置けなければ `公` に退避

となっています。

- 参照: `src/lib/shift-solver-fallback.ts:209-228`

このため、`infeasible` 時に「それっぽいが信頼しにくい表」が出るリスクは現実にあります。

この点については、既存メモの警戒感のほうが正しいです。

---

### 5. `allow_extra_off_days` は fallback だけの問題ではない

`allow_extra_off_days` は型と仕様書には存在します。

- 参照: `src/types/index.ts:28-29`
- 参照: `docs/shift-rules.md:43`

しかし、少なくとも今回見た `shift-solver-csp.ts` と `shift-solver-fallback.ts` では参照されていません。

したがって、これは

- 「fallback で無視される」

というより、

- **現行ソルバー系全体で未反映**

と整理するほうが正確です。

## Q1 への回答

### `off_target` ソフト化の評価

Codex の答え:

- **かなり有望**
- まず最初にやる価値が高い
- ただし、それだけで全問題が解決するとまでは言えない

おすすめは、

1. `off_target` を `GLP_FX` から外す
2. `off_pos / off_neg` を導入して偏差を目的関数に入れる
3. `must_pair` とシニア制約がどれだけ `infeasible` に寄与しているかを分けて測る

という順です。

Codex の感覚では、

- `off_target` は「最大の犯人候補」
- ただし「唯一の犯人」とまではまだ断定しない

です。

## Q2 への回答

### GLPK(MILP) と OR-Tools(CP-SAT) の実用差

Codex の整理は次の通りです。

#### 短期

今の規模なら GLPK でもまだ改善余地があります。

- スタッフ20〜30人
- 月31日
- 4値 one-hot

この規模だけで即限界、とは思いません。

#### 中長期

この問題の性質は CP-SAT のほうが相性が良いです。

特に差が出るのは次の点です。

##### 制約追加のしやすさ

- GLPK/MILP: 書けるが、補助変数や線形化の設計負担が増えやすい
- CP-SAT: 論理制約、再ified制約、違反ペナルティ設計が素直

##### infeasible 診断

- 今の GLPK 実装はほぼ「解けた/解けない」で終わっている
- どの制約を緩めれば通るか、を運用に乗せるなら CP-SAT 側が組みやすい

##### 解の質

- 今の目的関数は夜勤偏差しか見ていない
- 今後「希望はなるべく守る」「不公平を減らす」「例外コストを最小化する」が増えるなら CP-SAT のほうが伸ばしやすい

結論として、

- **今すぐ全部捨てる必要はない**
- **でも本命適性は OR-Tools 側**

です。

## Q3 への回答

### OR-Tools へ移る判断基準

Codex は、思想より運用指標で決めるのがよいと考えます。

まず TS/GLPK 側で最低限やるべき修正は3つです。

1. `off_target` のソフト化
2. 前月連勤持ち越しを consecutive 制約に反映
3. 求解前に単純な capacity check を入れる

この3つをやったうえで、次のどれかに当たるなら OR-Tools 移行を本格化してよいです。

- 妥当な入力でも `infeasible` が 10% を超える
- 30秒近い求解が常態化する
- 新要件が「ソフト制約の重み調整」「緩和候補提示」「説明可能な違反表示」に入る
- fallback を直し続けないと運用できない
- 制約追加のたびに既存ロジックとの衝突調整が必要になる

逆に、上の3点修正でかなり安定するなら、

- いきなり本番切替ではなく
- OR-Tools を shadow run で並走

がよいです。

## Q4 への回答

### Python + FastAPI + OR-Tools の現実的コスト

Codex の見積もりは次の通りです。

#### 基本実装だけ

- 2〜3週間

対象:

- API 契約
- 基本制約の CP-SAT 実装
- 単発求解
- Next.js 側との接続

#### 実運用に耐える形まで

- 5〜8週間

対象:

- 既存入力との整合
- shadow run
- TS版との比較基盤
- 差分分析
- ログ/監視
- エラー時挙動
- 切替判断

したがって、Claude Code の

- **6〜8週間**

という見積もりは、production 切替まで含むなら妥当です。

---

## Codex の最終スタンス

Codex の最終スタンスは次の通りです。

1. 既存メモの「本命は OR-Tools」は正しい
2. ただし、その前に `off_target` ソフト化は試すべき
3. それは「延命」ではなく、モデルとして自然な修正
4. ただし fallback 延命に投資しすぎるのは危ない
5. 次の一手は「GLPKを少し正す」+「OR-Tools shadow run 準備」の並走が最善

短くまとめると、

- **TS solver を完全に見限る段階ではまだない**
- **でも長期本命は OR-Tools でよい**
- **最初の再評価ポイントは `off_target` ソフト化後**

## Claude への返答用の一文要約

Codex の見解としては、「OR-Tools 本命」は妥当。ただし現行 GLPK モデルは `off_target` のハード等号が強すぎるので、まずそこをソフト化し、前月連勤持ち越しも直したうえで再評価するのが順当。fallback 延命より、GLPK の最低限是正と OR-Tools shadow run を並行するのがよい、という立場です。
