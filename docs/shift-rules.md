# シフトルール・制約 整理ドキュメント

> 作成目的: バグ修正の前提として「何が正しい動作か」を明文化し、実装と仕様のズレを可視化する。

## 1. シフトコード一覧

| コード | 意味 | 休日扱い | 上書き可否 |
|--------|------|----------|------------|
| `日`   | 日勤 | No       | 基本的に可 |
| `夜`   | 夜勤 | No       | フリーズ   |
| `明`   | 明け（夜勤翌日）| No | フリーズ |
| `公`   | 公休 | **Yes**  | 基本的に可 |
| `有`   | 有給 | **Yes**  | フリーズ   |
| `希休` | 希望休 | **Yes** | フリーズ  |
| `他`   | 特別休暇 | **Yes** | フリーズ |
| `''`   | 未割り当て | — | —       |

**休日カウント対象**: `公`, `有`, `他`, `希休` の 4 種類。

## 2. スタッフ属性

### 役職 (role)
| 値 | 意味 |
|----|------|
| `師長` | 病棟師長。シニア扱い |
| `主任` | 主任看護師。シニア扱い |
| `一般` | 一般看護師 |

**シニア判定**: `役職 === '師長' || 役職 === '主任'`

**シニアの夜勤回数**: 役職による特別な上限はなく、個人設定の `max_night_shifts` に従う。

### 主要フラグ

| フィールド | 説明 |
|-----------|------|
| `max_night_shifts` | 月間夜勤上限回数（0 = 夜勤不可） |
| `off_days_of_week` | 定休曜日（0=日〜6=土） |
| `off_days_constraint` | `'hard'`=厳守 / `'soft'`=可能なら守る |
| `off_on_holidays` | 祝日を定休にするか |
| `allow_extra_off_days` | `false` の場合、定休日以外は原則日勤。ただし目標休日数に未達の場合は追加公休が必要 |

## 3. グローバル制約 (ShiftConstraints)

| 制約 | フィールド | デフォルト |
|------|-----------|-----------|
| 夜勤最低人数/日 | `min_staff_per_shift['夜勤']` | 2 |
| 日勤最低人数/日（平日） | `min_staff_per_shift['日勤']` | 5 |
| 土日祝の最低日勤人数 | `min_staff_weekend` | 2 |
| 土日祝の最大日勤人数 | `max_staff_weekend` | 2 |
| お風呂の日 最低人数 | `min_staff_bath_day` | 平日日勤最低人数と同じ（未設定時）。設定画面で個別変更可能 |
| 連続勤務上限 | `max_consecutive_work_days` | 5 |
| 目標休日数（月） | `target_off_days` | その月の土日祝の数（不足は翌月繰越） |

## 4. 休日数ルール

### 目標休日数の考え方

各スタッフが当月に取得しなければならない休日数（= `staffTargetMap[id]`）は以下の合計：

```
staffTargetMap[id] = 前月繰越日数 + 当月の土日祝日数 + 有休取得日数
```

**例**: 繰越2日、6月の土日祝8日、有休1日の場合 → 目標 **11日**

- 当月の土日祝日数はスタッフ共通の基準（`target_off_days`）。
- 定休日が設定されているスタッフも、「定休日4日 + 他4日 = 月8日」の構成で同じ8日を目標とする。
- 有休（`有`）・希望休（`希休`）・特別休暇（`他`）はすべて休日扱いでカウントされる。
- 不足分は翌月に繰越（目標未達分は消化しなければならない）。

### `allow_extra_off_days = false` の場合

- 定休日（hard 制約）以外は原則日勤に充てる。
- ただし目標休日数（土日祝の数）に達していない場合は、追加で公休を付与する必要あり。
- 現実装では `staffTargetMap[id] = 定休日数 + 祝日数（hard のみ）` と算出しているが、
  これは「定休日で必ず取れる日数」のみを目標としており、追加配分が行われないケースがある。

### 繰越 (carryOver)
- 当月に目標休日数に届かなかった分（shortfall）を翌月に繰越。
- 翌月の `staffTargetMap` に加算される。
- 繰越があるスタッフは翌月の目標が増える。

## 5. シニアカバレッジルール（最重要制約の一つ）

**ルール**: 平日（土日・祝日以外）は必ず **シニア（師長または主任）が1人以上、日勤 (`日`) に入ること**。

### このルールが関係するケース

| 状況 | ルールの適用 |
|------|-------------|
| シニアが希望休・有給・公休 | 別のシニアが日勤に入らなければならない |
| シニアが夜勤 | 当日は夜勤＝日勤不可 → 他シニアが当日カバー必須 |
| シニアが明け | 明けは夜勤とセットのシフトのため日勤不可 → 他シニアが日勤必須 |

### 実装箇所

1. **Step B (夜勤)**: シニアを夜勤に入れる場合、翌日に他シニアがカバーできるか先読みチェック。
2. **Step C (日勤優先確保)**: 平日でシニアが日勤にいない場合、強制的に割り当て。
3. **Pass E-1 (公→日変換)**: 変換先が平日の場合、dayCount チェックはあるが**シニアカバレッジチェックなし**（現実装の潜在バグ）。
4. **Pass E-2 (日→公変換)**: `isSeniorRole(member.role)` の場合のみ他シニアのカバレッジを確認。
   - 非シニアの変換では**シニアカバレッジを確認しない**（現実装の課題）。
5. **Pass E-2b (シニアスワップ)**: シニア固有の日↔公スワップ時にカバレッジを確認。
6. **Pass E-swap**: `isSeniorRole(underStaff.role)` の場合のみカバレッジを確認。

## 6. ペア制約 (StaffPairConstraint)

### must_not_pair（同シフト禁止）

指定した2名が同じ日に同じシフト種別に入ることを禁止する。
主な用途: 準看護師同士の夜勤を禁止するなど。

チェック対象シフト:
- `shift_type_id === null` の場合: **日勤・夜勤の両方を禁止**（全シフト対象）
- `is_overnight === true` の場合: **夜勤のみ禁止**
- `is_overnight === false` の場合: **日勤のみ禁止**

**注意**: 両者が '公' や '希休' など休日扱いの場合は制約対象外。

**優先度**: CONSTRAINTS.md の制約優先度テーブルで優先度 6（ハード）に分類される。
現在の GLPK ソルバーはすべてのハード制約を同時に満たす解を探す。
他のハード制約（シニアカバレッジ等）と競合して解なしになった場合は `infeasible` を返す。
優先度の低い制約を自動緩和する仕組みは現時点では実装していない。

## 7. アルゴリズムフロー（Step A〜F）

### Step A: ハード制約の事前確定（フリーズ）

1. **A-1 前月持ち越し**: 前月末の夜勤者を当月1日の「明」・翌々日の「公」として確定（連続夜勤の場合は明→夜→明→公→公）
2. **A-2 休暇リクエスト**: 有給・希望休・特別休暇を確定（`frozenCells` に登録）
3. **A-3 定休日（ハード）**: `off_days_constraint === 'hard'` の定休日・祝日を公として確定（フリーズ）

4. **A-4 定休日（ソフト）**: `off_days_constraint === 'soft'` の定休日はフリーズしない。
   以下の方法で「可能な範囲での遵守」を実現する。
   - `allow_extra_off_days=false` かつ `max_night_shifts=0` のスタッフ:
     ソフト定休日**以外**の日に `no_extra_off` 制約（公休禁止）を追加する。
     これにより、公休の割当先がソフト定休日に自然と集まる（消去法による遵守）。
   - 上記以外のスタッフ（夜勤ありなど）:
     特別な制約は追加しない。目標休日数の目的関数による誘導のみ。
     日勤人数不足時はソフト定休日に日勤が割り当たることがある（例外）。

> **フリーズセル (`frozenCells`)**: Step B 以降で上書き不可。
> 以下が含まれる: A-1/A-2/A-3 で確定したセル、夜勤・明け・夜勤後公休

---

### Step B: 夜勤割り当て

**条件（候補スタッフ）:**
- `grid[id][dayIdx] === ''`（空きセル）
- 夜勤回数が `max_night_shifts` 未満
- 前日が夜勤・明けではない
- 2日前が夜勤ではない（通常パターン: 夜→明→公）。ただし連続夜勤パターン（夜→明→夜→明→公→公）は例外。夜勤回数が多いスタッフの場合、アルゴリズムが連続夜勤を選択することがある
- 連続勤務制約を超えない
- 翌日が空きまたは非フリーズ公
- 翌々日が空きまたは非フリーズ公

**割り当て後の処理:**
- 当日: `夜`（フリーズ）
- 翌日: `明`（フリーズ）
- 翌々日: `公`（フリーズ）

---

### Step C: シニア日勤優先確保（平日のみ）

- シニアが日勤にいない平日は、候補シニアを日勤に割り当て。
- ペア制約・連続勤務を考慮（緊急時は連続勤務を超過して警告）。

> **⚠️ 設計上の疑問**: 現在の順序は Step B（夜勤）→ Step C（シニア日勤）。
> ただし「シニアの日勤を先に固定してから夜勤を組む」方が、シニアカバレッジをより確実に確保できる可能性がある。
> Step B でシニアが夜勤に入ることで翌日の明けが確定し、Step C の日勤割り当て候補が減るという問題も起きている。
> → Step C を Step A の直後（Step B より前）に移す設計変更の検討余地あり。

---

### Step D: 一般日勤で最低人数を充足

**候補ソート順（優先度高→低）:**

1. ハード定休スタッフ（利用可能日が限られる）
2. **シニアは後回し**（Step C 済みのため過剰割り当てを防ぐ）
3. 公休数が多い順（多く休んでいるスタッフを先に使う）
4. 日勤数が少ない順（均等配分）

> **⚠️ 根本的な問題の所在**: シニアを後回しにすることで、
> シニアが公休になりやすく→目標より休日数が超過しやすい。
> 非シニアは日勤優先で割り当てられ→目標より休日数が不足しやすい。

---

### Step E: 休日数調整

#### Pass E-0: `allow_extra_off_days=false` スタッフ調整
- 非フリーズ・非定休日の `公` を `日` に変換。
- このスタッフは定休日のみ休みとする。

#### Pass E-swap: 過剰↔不足スタッフのスワップ
- 休日超過スタッフ（`公`）と休日不足スタッフ（`日`）を同日でスワップ。
- `dayCount` が変わらないため、最低人数制約を破らない。
- **チェック項目**:
  - overStaff が '公'→'日' になれるか（連勤制約）
  - overStaff が '日' に入るとペア制約違反か
  - underStaff がシニア・平日の場合、他シニアが日勤にいるか（カバレッジ維持）

> **現実装の制限**: underStaff がシニアの場合のみカバレッジチェックあり。
> overStaff がシニアでも underStaff が非シニアの場合、overStaff が '日' になることで
> シニアカバレッジが解消されるケースは**チェックなし**（不足）。

#### Pass E-1: 過剰休日スタッフの `公→日` 変換
- `offCount > staffTargetMap[id]` のスタッフの `公` を `日` に変換。
- **スキップ条件**: 週末、連勤制約オーバー、ペア制約違反、土日上限超過
- **⚠️ シニアカバレッジチェックなし**（改善余地あり）

#### Pass E-2: 不足休日スタッフの `日→公` 変換（余裕あり日のみ）
- `offCount < staffTargetMap[id]` のスタッフの `日` を `公` に変換。
- **スキップ条件**: `dayCount <= minRequired`（日勤が最低人数ちょうど）
- シニアの場合のみ他シニアのカバレッジを確認。
- **現状**: `dayCount > minRequired` の余裕がある日のみ変換可能。

#### Pass E-2b: シニア専用の `日→公` スワップ
- シニアが休日不足（`needed > 0`）の場合、非シニアとのスワップで解消。
- 非シニアの `公` → `日`、シニアの `日` → `公`（dayCount 維持）。
- シニアカバレッジのロールバックあり。

---

### Step F: 翌月繰越計算

- 各スタッフの `shortfall = staffTargetMap[id] - totalOffCount` を計算。
- 不足分を `carryOver` として翌月に引き渡す。

## 8. 既知のバグ・制限事項

### BUG-1: シニア過剰休日 / 非シニア休日不足

**現象**: シニアの実績休日数 > 目標、非シニアの実績休日数 < 目標

**根本原因**: Step D の候補ソートで「シニアは後回し」にするため、
シニアは公休になりやすく（休みすぎ）、非シニアは日勤が多くなる（休み少なすぎ）。

**E-1/E-2 で修正しきれない理由**:
- E-1（過剰→日勤変換）: 週末の `公` は無条件スキップ。週末に溜まった超過公休が変換されない。
- E-2（不足→公休変換）: `dayCount <= minRequired` の日はスワップなしでスキップ（簡略化版では）。
- E-swap: カバレッジチェックが完全でない場合に違反が発生する。

**状態（2026-05-06時点）**: 未修正。E-swap を追加したが完全な解決には至っていない。

### BUG-2: 非シニアが公休になることでシニアカバレッジが0になる

**現象**: シニアが希休等でフリーズ済みの平日に、非シニアの調整処理が走り
非シニアを公休に変換 → シニアカバレッジ = 0 という状態になり得る。

**原因**: E-2 の「非シニアの日→公変換」時にシニアカバレッジチェックがない。

**影響**: 平日でシニアが誰も日勤に入らない日が生じる。

### BUG-3（解消済み）: E-2 の swap ロジック削除による violation
- セッション中に E-2 の swap ロジックを誤って削除。
- 現在は E-2b（E-2 の後）でシニア専用スワップが復元されている。
- ただし非シニアの休日不足は E-2b では対応されていない。

## 9. 修正方針の整理（次のステップ）

### Option A: Step D のシニア優先度を調整
- シニアを最初に割り当てる（現在は最後）。
- メリット: 根本的な原因を修正
- リスク: シニアに日勤が集中しすぎる可能性。ペア制約違反の危険。

### Option B: E-swap の強化
- E-swap のカバレッジチェックを strengthening（overStaff がシニアかつ underStaff が非シニアの場合も考慮）。
- E-1 のシニアカバレッジチェックを追加。
- E-2 の非シニア変換時もシニアカバレッジを確認。

### Option C: ILP ソルバー（GLPK）側での修正
- フォールバックではなくメインの ILP ソルバーを使うよう条件を変更。
- 根本的な解決だが実装コストが高い。

## 10. テスト観点

| チェック項目 | 期待値 |
|-------------|--------|
| 各スタッフの実績休日数 | `staffTargetMap[id]` に近い（±1〜2日以内） |
| 平日のシニア日勤人数 | 1人以上 |
| 土日の日勤人数 | `min_staff_weekend` 以上 `max_staff_weekend` 以下 |
| 平日の日勤人数 | `min_staff_per_shift['日勤']` 以上 |
| 連続勤務日数 | `max_consecutive_work_days` 以下（原則） |
| 夜勤翌日 | 必ず `明` |
| 夜勤翌々日 | 必ず `公` |
| フリーズセル | Step B 以降で上書きなし |
| must_not_pair | 両者が同日に指定シフト種別にならない（shift_type_id=null の場合は日勤・夜勤両方） |
