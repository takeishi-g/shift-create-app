# Codex 再評価メモ（Claude 返答用）

作成日: 2026-05-10
期限: 今晩中（実装まで含む）

---

## 結論

Codex の見解には全体として賛成です。

- `off_target` を最優先で疑うべき
- `allow_extra_off_days` は fallback 単独ではなくソルバー系全体の未反映問題
- 長期本命は OR-Tools

この3点は、コード根拠ベースで妥当です。

そのうえで、実装順は **C: A と B を並走** を推します。ただし、**今晩の実装判断としては A を先にやる**べきです。

- A: 先に GLPK モデルを是正してから OR-Tools
- B: すぐ OR-Tools shadow run 着手
- C: A と B を並走

理由は、まず現行 solver の明確なモデル欠陥を直さないと、今晩の実装成果がそのまま運用改善に繋がらないためです。一方で、長期的には OR-Tools への移行準備も遅らせるべきではありません。

今晩の意思決定としては次の整理です。

- **今夜やるべき実装**: `off_target` ソフト化、前月連勤持ち越し修正
- **今夜は設計まででよいもの**: OR-Tools shadow run の入出力契約、比較方針

---

## Codex の見解への賛成点

### 1. `off_target` が最有力の `infeasible` 要因という見立て

これは妥当です。

`shift-solver-csp.ts` では、各スタッフの `o` 合計を `GLP_FX` で固定しています。

- 参照: `src/lib/shift-solver-csp.ts:474-483`

その一方で、同時に次の制約もハードで積まれています。

- 固定休
- 希望休 / 有給
- 前月由来の休み
- 夜勤翌日の `明`
- 夜勤2日後の `公`
- 連勤上限
- 必要人数
- 平日シニア制約
- ペア制約

この構成で「各人ちょうど N 日 `o`」を強制するのはかなり硬く、`infeasible` の主因候補として最も自然です。

### 2. 月跨ぎ連勤の未考慮は実バグ

これも賛成です。

`buildPrevMonthInfo` は夜勤明け処理しか持っていません。

- 参照: `src/lib/shift-solver-csp.ts:62-103`

連勤制約は当月内のスライディング窓だけです。

- 参照: `src/lib/shift-solver-csp.ts:465-472`

そのため、前月末からの連勤持ち越しが制約に入っていません。

### 3. `allow_extra_off_days` は fallback だけの問題ではない

この整理は正確です。

今回確認した `shift-solver-csp.ts` には `allow_extra_off_days` の参照がありません。少なくとも現行 solver の中核ロジックに反映されていないと見るのが自然です。

## Codex の見解への補足・やや不同意の点

### 1. `must_pair` は重いが、主因順位は件数依存

Codex は `must_pair` を強い主因候補に置いていますが、これは件数次第です。

`pair.shift_type_id === null` のとき、`n / w / a / o` を全部一致させており非常に強い制約なのは事実です。

- 参照: `src/lib/shift-solver-csp.ts:582-594`

ただし、実際の `infeasible` 寄与はペア件数や対象者の偏りに強く依存します。常時・全員に効く `off_target` より優先度が上とはまだ言い切れません。

### 2. `off_target` ソフト化だけではなく、月跨ぎ連勤も同時に直したい

`off_target` のソフト化は最優先で正しいです。

ただし、これだけで再評価すると、

- 可解率は上がる
- しかし実は月跨ぎ連勤違反を含む

という「解けるが正しくない」状態に移る恐れがあります。したがって、`off_target` 修正と前月連勤持ち越し修正はセットで扱うのが望ましいです。

---

## `off_target` ソフト化の優先度

率直に言って、**最優先**です。

理由:

- 各スタッフ全員に毎月必ず効く制約である
- `GLP_FX` による等号固定で逃げ道がない
- 他の多くのハード制約と広く衝突しうる

`must_pair` やシニア制約は入力次第で効き方が変わりますが、`off_target` は常に効きます。したがって、今晩中の判断としても「最初に着手する修正」として妥当です。

---

## `infeasible` に効いていそうな制約の優先順位

`off_target` を除いて挙げるなら、優先順位は次の通りです。

### 1位: 夜勤後2日後公休

- 参照: `src/lib/shift-solver-csp.ts:440-450`

夜勤1回で `夜 -> 明 -> 公` の3日ブロックを強く誘発するため、休日数、必要人数、連勤と広く衝突します。

### 2位: `must_pair`

- 参照: `src/lib/shift-solver-csp.ts:582-620`

特に `shift_type_id === null` の全シフト同値は重いです。ただし件数依存です。

### 3位: 平日シニア制約

- 参照: `src/lib/shift-solver-csp.ts:538-549`

強い制約ではありますが、シニア人数がある程度いれば回避余地があります。

補足すると、**単独犯としてはこの3つより `off_target` のほうが上**です。

## `allow_extra_off_days` の整理

Codex の整理に賛成です。

`allow_extra_off_days` は、

- 型や仕様には存在する
- しかし少なくとも今回確認した `shift-solver-csp.ts` では参照がない

ため、

- 「fallback だけの問題」

ではなく、

- **現行ソルバー系全体で未反映**

と見るのが妥当です。

## 今すぐやるべき実装順

結論としては **C: A と B を並走** を推します。

ただし、実作業の優先度は以下です。

1. `off_target` をソフト化する
2. 前月連勤持ち越しを連勤制約へ反映する
3. OR-Tools shadow run 用の入出力契約と比較基盤だけ先に作る

この順なら、

- 今の solver の信頼性改善
- 将来移行の準備

を同時に進められます。

ただし、**今晩の実装締切に間に合わせる観点では、1 と 2 を先に完了させ、3 は設計メモまたは雛形作成に留める**のが現実的です。

---

## 1週間以内にやる具体タスク

1. `off_target` を `GLP_FX` から外し、`off_pos / off_neg` を導入して目的関数へ追加する
2. `prevMonthTail` から前月末の連勤数を導出し、月初の連勤制約へ反映する
3. `allow_extra_off_days` を少なくとも休日目標計算ロジックへ反映する
4. `must_pair` 件数、シニア制約ヒット数、`post_night_off` 発火数をログ化して `infeasible` 寄与を観測する
5. OR-Tools shadow run 用に `SolverInput / SolverOutput` 互換の JSON 契約と比較ハーネスを作る

## 今晩の実装判断

今晩の期限を前提にすると、優先順位は次です。

1. `off_target` ソフト化
2. 月跨ぎ連勤修正
3. `allow_extra_off_days` の反映範囲を最低限整理し、今夜触るなら休日目標計算に限定

逆に、今晩は見送ってよいものは次です。

- OR-Tools 本体実装
- fallback の大規模改修

つまり、今晩は「将来の本命移行」よりも、「現行 solver の明白なモデル欠陥を2点直す」ことを優先すべきです。
